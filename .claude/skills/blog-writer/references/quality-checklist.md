# 内容质量标准与自检细则

## 质量六维标准

### 1. 深度
每个核心知识点必须覆盖四问：
- **是什么**：精确定义，避免模糊描述
- **为什么**：动机、设计取舍、历史背景
- **怎么做**：步骤化实施路径，含参数建议区间（如"连接池大小建议 CPU 核数 × 2–4"）
- **何时不用**：反模式、不适用场景、过度工程风险

### 2. 广度
文章必须覆盖以下维度（按深度级别裁剪）：

| 维度 | L1 入门 | L2 进阶 | L3 专家 |
|------|---------|---------|---------|
| 原理机制 | 概念介绍 | 机制拆解 | 源码/公式/证明 |
| 工程实现 | 示例代码 | 完整实现 | 生产级代码 |
| 运维监控 | 简述 | 关键指标 | 全链路可观测 |
| 风险控制 | 注意事项 | 失败模式 | 混沌工程 |
| 成本权衡 | 不要求 | 资源/时间成本 | TCO/ROI 分析 |
| 商业考量 | 不要求 | SLA/扩展性 | 架构演进路径 |

### 3. 准确性
- 术语使用遵循行业标准（如 CAP 不写成 CAP 理论三选二而不解释）
- 结论有依据（官方文档、论文、实测数据），不出现"一般来说"等模糊措辞
- 前后自洽：第 5 节说"推荐 A 方案"，第 8 节不能说"A 方案有严重缺陷"而不解释

### 4. 可执行性
- 给出具体参数建议区间，不只说"根据实际情况调整"
- 提供命令行/配置文件可直接复制使用
- 步骤有明确的完成标志（"执行后看到 `xxx` 输出表示成功"）

### 5. 教学性
- 从简单概念出发，逐层引入复杂度
- 每个 H2 节结尾可加一句小结（"本节要点：..."）
- 类比优先：用读者熟悉的概念解释陌生概念

### 6. 商业可用性
至少涉及：
- **SLA 影响**：该方案对可用性/延迟的影响
- **可维护性**：团队学习曲线、文档质量、社区活跃度
- **扩展性**：未来流量/功能增长时的演进路径
- **团队协作成本**：需要什么技能？上手时间？

## 写作规范

### 标题层级
```
# 文章标题（H1，只出现一次）
## 一级章节（H2）
### 子章节（H3）
#### 细分（H4，慎用）
```

### 列表使用规则
- **要点列表**（无序）：并列的、顺序无关的内容
- **编号列表**（有序）：有先后顺序的步骤、流程
- **定义列表**：术语解释（用加粗 + 冒号格式）

### 强调规范
- `代码`：命令、文件名、变量名、配置项
- **加粗**：关键概念首次出现、重要警告
- *斜体*：术语首次定义、外文词汇
- > 引用块：重要结论、警告、注意事项

### 表格规范
对比表必须包含的列：方案名、核心优点、核心缺点、适用场景、实现成本（低/中/高）、风险等级（低/中/高）。

## 自检执行清单

输出前逐项确认，不通过则重写对应段落：

### 完整性检查
- [ ] 标题包含核心关键词
- [ ] 摘要 100–200 字，含问题/方法/结论
- [ ] 所有 14 个骨架模块均已覆盖（或有充分理由合并）
- [ ] 目录与实际章节一致

### 逻辑性检查
- [ ] 背景中提出的问题，在方案设计中得到解答
- [ ] 对比表中的结论与正文推荐一致
- [ ] 没有悬空的结论（有主张 → 有论据）
- [ ] 没有自相矛盾的建议

### 实用性检查
- [ ] 代码示例可复制运行（或明确标注需要哪些环境）
- [ ] Checklist 条目可直接执行，不是抽象建议
- [ ] 排障清单有具体的定位命令/工具
- [ ] 至少 3 个实践建议含具体参数/阈值
- [ ] 至少 3 个风险提示含修复方案

### 读者匹配检查
- [ ] L1：无预设知识，所有概念均有解释
- [ ] L2：假设读者了解基础，重点在机制和工程实践
- [ ] L3：可跳过基础定义，深入源码/公式/边界条件

### 商业可用性检查
- [ ] 提及了 SLA 或可用性影响
- [ ] 提及了方案的维护成本
- [ ] FAQ 涵盖了"什么时候不该用这个方案"

## 常见写作陷阱

1. **概念堆砌**：列了 10 个术语但没说怎么用 → 解决：每个概念后接"在本文场景中，它负责..."
2. **代码孤岛**：贴了代码但没说怎么集成 → 解决：补充"将以下代码放入 `xxx` 文件的 `yyy` 方法中"
3. **假装对比**：对比表只说优点，缺点写"无明显缺点" → 解决：每个方案必须有至少 1 个真实缺点
4. **虚假可执行性**：Checklist 写"确保系统稳定性" → 解决：改写为"将 `max_connections` 设置为不超过 CPU 核数 × 4"
5. **忽略失败路径**：只描述正常流 → 解决：在"方案设计"中明确加入异常流/降级策略
