# Blog Writer

## Overview

产出专业技术博客/文档，覆盖原理、工程实现、运维监控、风险控制、成本权衡，既可作学习教材也可直接用于企业内外部发布。

## 输入参数

| 参数 | 必填 | 说明 |
|------|------|------|
| 主题 | ✅ | 如"Redis 缓存一致性设计" |
| 目标读者 | ✅ | 初级/中级/高级 + 技术背景描述 |
| 文章类型 | ✅ | 原理解析 / 实战教程 / 方案对比 / 案例复盘 / 性能调优 |
| 技术栈 | ✅ | 语言、框架、中间件、部署环境 |
| 深度级别 | ✅ | L1 入门 / L2 进阶 / L3 专家 |
| 篇幅范围 | ☐ | 短文 1500–2500 / 长文 3000–6000 / 深度专题 8000+ |
| 输出语言与风格 | ☐ | 默认：中文，专业严谨、可读性强 |

**若用户仅给主题**，按默认参数生成：中级读者 + 实战教程 + L2 深度 + 长文 3000–6000 字。

## 写作工作流

### 第一步：解析输入，确认参数

收到任务后，先列出已确认的参数。若必填参数缺失，询问用户补充；若用户指示直接生成，按默认值填充并在文章开头标注。

### 第二步：构建文章骨架

严格按照固定骨架结构（见 `references/article-structure.md`）展开，**不得省略任何模块**（极短文章可合并，但须保留要点）。

### 第三步：内容深度要求

每个模块必须回答：
- **是什么**（定义 + 术语首次出现给一句话解释）
- **为什么**（动机、场景、痛点）
- **怎么做**（步骤化实施路径 + 参数建议区间）
- **何时不用**（反模式、禁忌场景）

### 第四步：必含元素（逐项核查）

- [ ] 至少 **3 个实践建议**（可执行，有具体参数/阈值）
- [ ] 至少 **3 个风险提示**（症状 → 原因 → 修复）
- [ ] 至少 **1 个对比表**（方案对比或技术选型）
- [ ] **代码/配置示例**（可运行，含注释，标明语言）
- [ ] **最佳实践 Checklist**（条目可直接勾选执行）
- [ ] **FAQ**（不少于 5 问，覆盖常见误区）

### 第五步：输出前自检（逐项通过才能输出）

1. **完整性**：结构模块是否齐全？
2. **逻辑性**：论证链路是否闭环？结论有据可查？
3. **实用性**：是否提供了工程落地路径？还是仅有概念堆砌？
4. **读者匹配**：内容难度是否符合目标读者？
5. **商业可用性**：是否关注了 SLA、可维护性、扩展性、团队协作成本？

若任一项不满足，**自动重写对应段落后再输出**，不提示用户。

## 输出格式

默认输出 Markdown，包含：

```
# 标题
> 摘要（100–200 字）

## 目录
## 背景与问题定义
## 核心概念与原理
## 方案设计与实现
## 代码与配置示例
## 多方案对比
## 性能与可靠性
## 安全与合规
## 常见误区与排障清单
## 最佳实践 Checklist
## 总结与扩展阅读
## FAQ
```

## 深度级别行为差异

| 级别 | 代码量 | 原理深度 | 运维/监控 | 商业考量 |
|------|--------|----------|-----------|----------|
| L1 入门 | 简单示例 | 概念为主 | 简述 | 不要求 |
| L2 进阶 | 完整可运行 | 机制拆解 | 覆盖 | 成本/扩展 |
| L3 专家 | 生产级 | 源码/公式 | 全链路 | SLA/架构权衡 |
